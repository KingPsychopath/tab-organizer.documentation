Tab Organizer Help/FAQ

What's the difference between a selected tab and a focused tab?

A selected tab is one that you have selected, for instance with Ctrl or Shift. These show up as green within Tab Organizer.

A focused tab is one that is currently focused within Chrome. These show up as tan/orange within Tab Organizer.

Does Tab Organizer support keyboard accessibility?

Absolutely! I put quite a bit of work into making sure that it is possible to do everything with the keyboard only.

Some things (like moving tabs) are more difficult with the keyboard, but it's still possible.

What are special queries?

Special queries are designed to make it easier to find certain types of tabs.

As an example, you can use the same:url special query to find all tabs that have the same URL.

How do I use special queries?

Just type them into the search box. You can also combine special queries and advanced syntax.

For instance, same:url | google will find all tabs that have the same URL, and also the tabs that contain the text google.

What are the different special queries?

  • has:macro

    Shows the tabs that match one or more macros.

  • intitle:

    Shows tabs whose title matches the supplied text. For instance, intitle:caturday would show tabs that have caturday in their title.

  • inurl:

    Shows tabs whose URL matches the supplied text. For instance, inurl:google.com would show tabs that have google.com in their URL.

  • is:bookmarked

    Shows the tabs that are bookmarked within Chrome.

  • is:broken

    Shows the tabs that "broken", like a 404 error or the server not responding. The detection isn't perfect, so it may match a tab that isn't broken, and it might not match a tab that is broken.

  • is:child

    Shows the tabs that are children of another tab (indented).

  • is:favorited

    Shows the tabs that have been favorited. You can favorite/unfavorite a tab by clicking on the star on the left side of the tab.

  • is:image

    Shows the tabs that are images.

  • is:pinned

    Shows the tabs that are pinned within Chrome.

  • is:selected

    Shows tabs that have been selected (for instance, with Ctrl or Shift).

  • last:moved

    Shows the tabs that were last moved. This includes tabs that were moved by a macro.

  • same:domain

    Shows tabs whose domains are the same. As an example, two or more tabs on the www.google.com domain would match. If a domain is used by only a single tab, that tab will not be shown.

  • same:path

    Shows tabs whose domain + path is the same. This is useful for finding similar tabs.

  • same:title

    Shows tabs whose titles are the same. This is useful for finding duplicate tabs.

  • same:url

    Shows tabs whose URLs are the same. This is useful for finding duplicate tabs.

    Note: this excludes trailing slashes and hash fragments. That means that http://foo/bar/ and http://foo/bar#qux are treated as the same URL.

  • seen:url

    Shows tabs that you have seen (within Chrome), based on their URL.

  • will:move

    Shows the tabs that will be moved, if you use Apply all macros.

  • window:

    Shows tabs that are within a certain window. For instance, window:5 would show all the tabs in the window named 5.

    You can also use a comma to easily show multiple windows at once: window:3,5,9

    And you can use a dash to show windows in sequence: window:1-5

    Note: this works with custom names too. If you rename a window foo, you can show it with window:foo

    Note: there is a special value called focused that will show the window that is focused within Chrome.

    For instance, window:focused will show the focused window, and window:1-5,focused will show windows 1 through 5, and the focused window.

What's the point of inurl: and intitle:?

inurl: is useful when searching for things that are only found in URLs, like hash fragments or URL queries:

  • inurl:#
  • inurl:?
  • inurl:https://

intitle: is useful when you want to exclude things that normally appear in the URL. For instance, let's say you have a tab whose title contains the phrase http.

If you typed in http it would probably match most of your tabs, because URLs often start with http:// On the other hand, you can use intitle:http to find the correct tab.

Another example is searching for the phrase com. Websites often have that in the URL, so using intitle:com will filter those out.

You mentioned advanced syntax, what's that?

Advanced syntax is a way to combine queries together to make your searches even more accurate, helping you to find the exact tabs you want.

If you have used Google Search, you may already know that it supports advanced syntax. Tab Organizer is very similar. Here is a list of the advanced syntax supported by Tab Organizer:

  • -

    Excludes an item from the search. For instance, -google would show all tabs that do not contain the text google.

    This works with special queries as well. For instance, you can use -window:5 to show every window except the window named 5.

  • ,

    This only works with special queries. It's an easier way to write OR.

    For example, the query same:url,title is the same as same:url | same:title

    And the query inurl:google,wiki is the same as inurl:google | inurl:wiki

    Note: this only works within one prefix. That means you can't do things like has:macro,is:pinned. You'll need to use has:macro | is:pinned. However, you can do things like this: has:macro | is:pinned,image

  • Space

    Using a space is the same as logical AND. This narrows your search to be more specific.

    For instance, wikipedia chrome would show tabs that match both wikipedia and chrome at the same time.

  • OR

    This adds items to the search. For instance, google OR youtube would show tabs that match either google or youtube.

    You can also use the | character, like this: google | youtube

  • ( )

    Ordinarily, AND has higher precedence than OR, but you can use parentheses to override that, like so: foo (bar | qux)

    You can also use them to do some pretty complicated stuff. For instance, -(foo | bar) is equivalent to -foo -bar

  • " foo "

    This is a shortcut for r/\b foo \b/i

    That means "foo" is the same as r/\bfoo\b/i

    This is used to find a word, while not allowing sub-word matches. For instance, "foo" would match foo, foo bar, or bar foo, but would not match foobar.

  • r/ regexp / [flags]

    This gives you the full power of JavaScript's regexp engine.

    For instance, r/^foo(bar)?$/ would find tabs that match foo or foobar, but the word must occur at the start and end. That means it would not match foo bar, barfoo, or bar foo. It is also case-sensitive, so it would not match Foo or fooBaR.

    If you want it to be case-insensitive, you can place the i flag after the closing slash, like this: r/foo/i Currently, the only flag that is supported is i.

    Regexps are usually too complicated and clunky to search with, but they can be very useful when used with macros.

You can combine the above as much as you like. For instance, same:url google | window:bar -inurl:youtube would show the tabs that have the same URL and match google, but also the tabs that are in the window named bar, but not those that have youtube in their URL.

What has higher precedence: AND or OR?

AND has higher precedence. This can be overriden with parentheses (see above). That means:

  • foo bar | qux

    (foo AND bar) OR qux

  • foo | bar qux

    foo OR (bar AND qux)

  • foo bar | qux corge

    (foo AND bar) OR (qux AND corge)

What are macros?

Macros let you do things such as move tabs into a window, close tabs, etc. Although you can do those things manually, macros let you do it much faster and easier. This is especially useful if you do the same actions repeatedly.

How do I make macros?

First, open the options page. On the left side, select the Macros category. Use the New macro button to create a macro. A macro is composed of three parts:

  1. an action

    This describes what the macro does.

  2. a search query

    This describes what tabs will be affected.

  3. an optional window name

    This specifies a single window within Tab Organizer. It must match exactly, including case sensitivity. If the window doesn't exist, it is created automatically. Leaving it blank means "new window".

What are the different macro actions?

  • ignore

    This causes the tabs to be ignored by macros. This only affects macros lower in the list, not higher.

  • require

    This action does two things. First, it takes tabs matching the search query, and moves them into the specified window. Second, it takes tabs that do not match the search query, and moves them out of the specified window.

    This allows you to divide tabs into windows based on their content. For instance, you may want all wikipedia tabs to be placed into the window named Wikipedia, all the youtube tabs placed into the YouTube window, etc.

  • move

    This is like require except it's one-way: it moves tabs into the window, but doesn't move non-matching tabs out. You can use this to collect many different kinds of tabs into a single window.

  • close

    This closes the matching tabs. You can't specify a window when using this action.

Okay, but how do I trigger the macro?

Within Tab Organizer, select the Menu button in the upper-left corner. Now select the Macros... submenu, which shows a list of all your macros.

You can trigger a single macro by selecting it, but there is also an Apply all macros item, that triggers all the macros in sequence.

Unlike triggering a single macro, Apply all macros takes all the macros into account. Macros higher in the list have higher priority, and will override macros that are lower in the list.

As an example, suppose you have these two macros:

  1. require foo in "Foo"
  2. require bar in "Bar"

What if you have a tab that matches both foo and bar? It will be moved into the window Foo, because that macro is higher in the list. This gives you control over how to handle these kinds of conflicts.

Can I see some examples of macros, and what they do?

  • close inurl:chrome://newtab/

    This closes all the New Tab pages.

  • require intitle:r/^File:/ | is:image in "Images"

    This makes sure that the window Images contains tabs that are images. This includes Wiki tabs that are wrappers around images.

  • require is:broken in "~Error"

    This tries to find sites that are broken, and moves them into their own window to make it easy to find them.

  • require inurl:r/google.com\/(?:search|images|imgres)/ in "~Search"

    Usually, I use Google to search for something, then end up leaving the search results page open even when I no longer need it. This macro moves them into their own window, making it easy to manually close them.

  • require wikipedia in "Wikipedia"

    I don't know about you, but I tend to read Wikipedia a lot, and end up with Wikipedia tabs scattered all over the place. This macro moves them into a single Wikipedia window, thus causing less clutter.

  • require inurl:youtube.com in "YouTube"

    This moves all the YouTube tabs into the YouTube window.

I use many others, but they tend to fit into my lifestyle and the kinds of tabs that I use. Make your own and experiment to find macros that work good for you.

Can you explain usage tracking?

Tab Organizer uses Google Analytics, which is also used by many other websites. Google Analytics tracks many things, such as browser, screen resolution, time spent on the site, etc.

However, I'm not interested in most of the information that Google Analytics gives me. I mostly use it to track the options page. This lets me see what settings are popular, so I can reorder them or change the defaults.

You can disable usage tracking in the options page, in the Privacy category. It will tell Google Analytics that you have disabled usage tracking. This lets me see how many users have disabled it. Nothing else is tracked in Tab Organizer after you disable usage tracking.

What things are tracked?

These things are tracked, and cannot be disabled in Tab Organizer:

  • How many users have visited the FAQ page.
  • How many users have visited the recent changes/changelog page.
  • How many users have installed Tab Organizer.
  • How many users have visited the bugs/feature requests page.
  • What issues are most visited.

These things are tracked, but can be disabled:

  • How frequently users reload Tab Organizer (this usually means restarting Chrome).
  • How frequently users open the popup.
  • How frequently users open the options page.
  • What options users choose in the options page.
  • How many windows and tabs the user has.
  • How many favorites and macros the user has.
  • The version of Tab Organizer that is currently installed.
  • The type of user. Currently, this is either "normal" or "developer".

However, we do not track the following:

  • Title or URL of any tab, except the ones mentioned above.
  • Search queries.
  • Window names.

Why is usage tracking enabled by default?

Simply because the information provided is anonymous, and has already been useful for improving Tab Organizer. Unfortunately, if it was disabled by default, I think most users would not enable it, even if they didn't have a problem with usage tracking.

I think that the information that Tab Organizer tracks is not personal, and that most users would not have a problem with me using this information to help make Tab Organizer better for everybody.

However, if I'm wrong, and most users would in fact object to usage tracking, then I would seriously consider having it off by default.

How has usage tracking been beneficial?

When Tab Organizer first got keyboard shortcut support, the default was Ctrl + Shift + O. Unfortunately, this shortcut is difficult for QWERTY users to use, and might be mistaken as Ctrl + O (which means Open in Chrome).

By using Google Analytics, I was able to see what keyboard shortcuts were popular. I didn't like most of the popular ones, but one in particular I liked: Ctrl + Shift + E.

This is easy to use for both QWERTY and Dvorak users, and I am unaware of any conflicts with Chrome. If I had not used usage tracking, the default might still be Ctrl + Shift + O, despite it being worse.

As another example: in the options page, the Keyboard category used to be at the top, with Theme being second. By using Google Analytics, I noticed that Theme was more popular, so I changed Theme to be the first category.

Usage tracking is also great for experimental features, which must be enabled in the options page. By seeing how many users use experimental features, I can make a more educated decision about whether to support them or not.
